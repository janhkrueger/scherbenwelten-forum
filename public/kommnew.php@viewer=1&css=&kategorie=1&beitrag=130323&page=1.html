Kapo el Ligno
Hier ein konstruktiver Vorschlag:
- vor jedem Update einen Validierungslauf auf einem Testsystem (natürlich wird es nicht möglich sein, lastbedingte Probleme damit aufzudecken, aber prinzipbedingte Probleme wie schieflaufende Stadtrudnen etc. - problemlos)

- Updates grundsätzlich nur kurz nach einem Backuplauf, und nicht "kurz vor dem nächste", damit im Falle eines notwendigen Rollbacks nicht die Daten eines kompletten Tages weg sind. Man kann auch über eine Änderung der Backupstrategie nachdenken, die ein zurückfahren zu einem beliebigen Zeitpunkt ermöglichen würde (solange nur logische Inkonsistenzen vorhanden sind, keine physischen). Ob MySQL das kann, weiß ich nicht, Oracle kann das.

- eine Checkliste aufstellen (im Prinzip dieselbe wie beim o.g. Validierungslauf), die sofort nach einen Update abgearbeitet wird und sicherstellen soll, dass die "testbaren" Szenarien weiterhin funktionieren, wie z.B. normales Laufen, normales Kämpfen, Armeebewegungen, Stadtrunden, Produktion etc., mit den gängigen Browsern (IE8, IE9, aktueller FF, Chrome, Opera) und Betriebssystemen (WinXP, Win7, Linux, MacOS) - die Auswertung der Logfiles etwa mit Webalizer ermöglicht einen guten Aufshcluß darüber, was nennenswert vertreten ist und womit nut 1-2 Leute surfen

- last but not least: für planbare Updates ("wie wollen den Code aufräumen und diverse Bugs fixen...") sollte man das deutlich früher ankündigen.

Hi,

das ist kein Bug, das ist leider eine Einstellung im Firefox (wie vermutlich auch im Chrome). Seit neuestem werden bei Firefox alle JS Events im Hintergrund, die öfter als alle 1000ms ausgeführt werden pausiert. Man kann das aber Einstellen wenn man will:

1. In der Adressleiste "about:config" eintippen
2. die Einstellung "dom.min_background_timeout_value" suchen
3. Dort den Wert auf 50ms oder 100ms runterdrehen.

Aber man sollte vorsichtig damit sein, da dieser Wert die Performance des kompletten Firefox beeinflusst. Also auf eigene Gefahr
